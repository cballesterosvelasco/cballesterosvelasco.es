Working modes

Early value

When working alone for OpenSource projects or for time/resource limited projects, I usually start creating a low quality spike. No testing. Just experimenting and getting feedback for me and my clients. Then usually iterate from there or just throw it away since I have not invested too much time and that's ok.

My philosophy here is to provide value fast. Not investing too much time at the beginning. Even when limiting some software design principles here for the sake of prototyping. I keep the code KISS and DRY as they are very important to me for later stages.

If the project still matters after some time, I usually start creating a test suite and later refactoring it. Since is should have be a small amount of code, without overengineering it is usually an easy step. And it will be as high-quality as the project deserves.

Quality from the beginning

On already existant codebases and/or where time is not a priority, but it has to be solid-rock. I usually work different.

Before starting, I ensure that I know what I really have to do validating with my client. When covering areas where I'm not experienced, I usually invest some time before starting familiarizing with the problem in a technical point of view.

I follow the TDD methodology. So start creating tests covering the cases I have to cover priorizing by value or requirements.

I refactor frequently trying to keep it as good as I can with the information I have at that moment. I really appreaciate a good IDE and tooling for this step.
